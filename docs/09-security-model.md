# Routely Security Model

Version: 0.1  
Status: Draft

## English

Routely executes code and manages production infrastructure, so the security model must be explicit from day one.

## Indonesia

Routely menjalankan code dan mengelola infrastructure production, jadi model security harus jelas sejak awal.

## 1. Trust Boundary

Trusted:

- Server owner/admin.
- Repositories explicitly connected by the admin.
- Local workspace controlled by the user.

Untrusted or semi-trusted:

- Pull request code.
- Public templates from future marketplace.
- Webhook payloads until verified.

## 2. Authentication

MVP:

- Local dashboard uses local token or first-run token.
- Production dashboard requires admin login.
- GitHub App used for repository access.

Post-MVP:

- 2FA.
- Multi-user/team support.
- Fine-grained permissions.

## 3. Secrets

- Store secrets encrypted where practical.
- Hide secret values in dashboard after creation.
- Redact secrets from logs.
- Never export secrets by default.

## 4. Network Defaults

- Production app containers should bind to internal Docker networks.
- Public access should happen through proxy only.
- Databases are internal-only by default.
- Public database exposure must require explicit confirmation.

## 5. Webhook Security

- Validate GitHub webhook signatures.
- Ignore unsupported events.
- Deduplicate delivery IDs.
- Rate limit webhook endpoint.

## 6. Command Execution Risk

Routely may execute build and start commands from repositories. This is inherently powerful.

MVP rule:

- Only deploy repositories explicitly connected by the admin.
- Show build commands before first deploy when generated by presets.
- Run production builds inside containers where possible.

## 7. Backups

- Backup files should not be publicly served.
- Backup retention should be configurable.
- Backup restore should require explicit confirmation.
