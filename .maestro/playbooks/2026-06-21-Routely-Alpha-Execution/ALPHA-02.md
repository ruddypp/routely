# Routely Alpha Phase 02: VPS Demo Hardening

Task-based playbook: each checkbox task must be executable in a fresh agent context. Route VPS, DNS, credential, and permission blockers to Routely Lead. Do not guess external access details.

- [ ] Backend: harden the one-VPS Dockerfile deploy path. Read `AGENTS.md`, `docs/01-alpha-plan.md`, `docs/03-demo-acceptance-plan.md`, `docs/05-architecture.md`, `docs/06-interfaces.md`, `docs/07-security-and-risks.md`, `docs/09-current-status.md`, and relevant CLI/daemon/package code. Verify and fix `routely server init`, `routely server doctor`, production data directory behavior, admin-token auth for mutation paths, Dockerfile deploy phases, deployment log persistence, and failure phase reporting. Add or update focused tests for server readiness, auth rejection, deploy state transitions, Docker helper behavior, and log persistence where code changes. Run relevant checks including `npm run lint`, `npm run test --workspace apps/cli`, `npm run build --workspace apps/cli`, and `node --check apps/daemon/src/server.js` if daemon code is touched. Commit only this task's files if verification passes.

- [ ] Backend: harden domain, DNS, proxy, and HTTPS state for the VPS demo. Read `docs/03-demo-acceptance-plan.md`, `docs/05-architecture.md`, `docs/06-interfaces.md`, `docs/07-security-and-risks.md`, and relevant `packages/proxy`, daemon, CLI, and DB code. Ensure domain add/verify behavior is honest, DNS verification uses configured server public IP where possible, generated Traefik-compatible routes target the latest successful Dockerfile deployment, and TLS state does not claim certificate success from config generation alone. Add or update focused tests for hostname validation, DNS verification, proxy route generation, and TLS status semantics where code changes. Run relevant checks and commit only this task's files if verification passes.

- [ ] Frontend/UIUX: make VPS demo dashboard surfaces real-data backed. Read `AGENTS.md`, relevant Next.js docs in `node_modules/next/dist/docs/`, `docs/03-demo-acceptance-plan.md`, `docs/06-interfaces.md`, and current dashboard/API route code. Ensure server readiness, deployments, deployment logs, domains, DNS/proxy/HTTPS state, health, and failure summaries are visible from real `/api/*` data. Unsupported production actions must be hidden, disabled, or marked deferred. Add or update web tests for touched route handlers or UI state. Run `npm run lint`, `npm run test --workspace apps/web`, and `npx tsc --noEmit --project apps/web/tsconfig.json` when web code is touched. Commit only this task's files if verification passes.

- [ ] QA E2E: create a disposable VPS smoke report under `docs/qa/`. Read `docs/03-demo-acceptance-plan.md` and route VPS/DNS/credential blockers to Routely Lead. Record provider, OS image, Node/npm, Docker, public IP, DNS provider assumptions, exact commands, expected/actual behavior, and evidence for `routely server init`, `routely server doctor`, Dockerfile deploy, domain add/verify, and HTTPS/proxy state. Include concrete Backend/Frontend bug instructions for blockers. Do not edit production code. Write the QA report and hand it to Routely Lead; do not commit the QA report yourself.

- [ ] Security: review VPS trust boundaries and write findings under `docs/security/`. Read `docs/07-security-and-risks.md` and `docs/03-demo-acceptance-plan.md`. Review production auth, admin token handling, Docker/Compose/proxy exposure, env/secrets exposure, backup file assumptions, DNS/HTTPS truthfulness, and dashboard access assumptions. Write findings with severity, evidence, and remediation owners. Do not edit production code unless explicitly assigned. Write the Security report and hand it to Routely Lead; do not commit the Security report yourself.

Human review after phase: Routely Lead/PM should confirm VPS demo readiness before GitHub auto-deploy work.
