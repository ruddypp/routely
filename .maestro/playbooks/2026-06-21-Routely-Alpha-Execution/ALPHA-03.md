# Routely Alpha Phase 03: GitHub Demo Hardening

Task-based playbook: each checkbox task must be executable in a fresh agent context. Route GitHub App, repo, webhook, credential, and permission blockers to Routely Lead.

- [ ] Backend: harden GitHub push-to-deploy behavior. Read `AGENTS.md`, `docs/01-alpha-plan.md`, `docs/03-demo-acceptance-plan.md`, `docs/05-architecture.md`, `docs/06-interfaces.md`, `docs/07-security-and-risks.md`, `docs/09-current-status.md`, and relevant GitHub/daemon/CLI/DB/deploy code. Preserve `X-Hub-Signature-256` validation, delivery dedupe, configured repo/branch filtering, app-to-repo mapping, and reuse of the Dockerfile deployment flow. Ensure deployment records expose useful repo, branch, commit SHA, delivery ID, status, failure phase, and logs where available. Add or update focused tests for signature validation, branch filtering, dedupe, mapping, deployment creation, and failure log persistence where code changes. Run relevant checks and commit only this task's files if verification passes.

- [ ] Frontend/UIUX: make GitHub diagnosis flow obvious and real-data backed. Read `AGENTS.md`, relevant Next.js docs in `node_modules/next/dist/docs/`, `docs/03-demo-acceptance-plan.md`, `docs/06-interfaces.md`, and current dashboard/API route code. Show GitHub connection status, configured repo/branch, latest delivery, latest deploy, ignored/failing events, commit metadata where available, failure phase, and path to deployment logs. Avoid implying preview deployments or unsupported GitHub features exist. Add or update tests for touched route handlers or UI state. Run web checks and commit only this task's files if verification passes.

- [ ] QA E2E: create a GitHub auto-redeploy smoke report under `docs/qa/`. Read `docs/03-demo-acceptance-plan.md`. With Routely Lead-provided GitHub App/repo/branch access, run one successful push and one intentionally broken push. Record webhook delivery IDs, commit SHAs, exact commands or dashboard steps, expected/actual deploy behavior, ignored branch behavior if tested, and log availability. File concrete Backend/Frontend bug instructions for any blocker. Do not edit production code. Commit only the QA report if complete.

- [ ] Security: review GitHub webhook and deploy trust boundaries under `docs/security/`. Read `docs/07-security-and-risks.md` and `docs/03-demo-acceptance-plan.md`. Review signature validation, replay/dedupe behavior, repo/branch authorization assumptions, secret exposure in logs/API/dashboard, installation token handling assumptions, and untrusted GitHub metadata rendering. Write findings with severity, evidence, and remediation owners. Do not edit production code unless explicitly assigned. Commit only the security report if complete.

Human review after phase: Routely Lead/PM should confirm GitHub demo readiness before release docs.
