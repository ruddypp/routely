# Routely Alpha Phase 01: Local Demo Hardening

Task-based playbook: each checkbox task must be executable in a fresh agent context. Stay within Routely repo conventions, preserve unrelated dirty worktree changes, and commit only files changed for the task after verification passes.

- [ ] Backend: harden and verify the local demo path. Read `AGENTS.md`, `docs/00-product-brief.md`, `docs/01-alpha-plan.md`, `docs/03-demo-acceptance-plan.md`, `docs/05-architecture.md`, `docs/06-interfaces.md`, `docs/09-current-status.md`, and relevant CLI/daemon/package code before editing. Ensure a clean workspace can run `routely init`, register three local command apps and one Compose-backed database service, run `routely`, inspect `routely ps`, inspect `routely logs <app>`, and stop with `routely down`. Preserve workspace-root separation and same-origin dashboard API rules. Add or update focused tests for any changed CLI, daemon, driver, config, dependency-ordering, port-conflict, or log behavior. Run relevant checks including `npm run lint`, `npm run test --workspace apps/cli`, `npm run build --workspace apps/cli`, and any targeted daemon syntax checks for touched files. Commit only this task's files with a concise message if verification passes.

- [ ] Frontend/UIUX: make the local dashboard demo-critical flow honest and real-data backed. Read `AGENTS.md`, `docs/01-alpha-plan.md`, `docs/02-team-execution-plan.md`, `docs/03-demo-acceptance-plan.md`, `docs/05-architecture.md`, `docs/06-interfaces.md`, and relevant Next.js docs under `node_modules/next/dist/docs/` before editing `apps/web`. Ensure overview, apps/services, and logs views show real daemon/API data for the local demo; browser code must use same-origin `/api/*` only. Hide, disable, or mark unsupported controls as deferred. Add or update route-handler/UI tests for changed dashboard behavior. Run `npm run lint`, `npm run test --workspace apps/web`, and `npx tsc --noEmit --project apps/web/tsconfig.json` when web code is touched. Commit only this task's files if verification passes.

- [ ] QA E2E: create a fresh local demo smoke report under `docs/qa/`. Read `docs/03-demo-acceptance-plan.md`, `docs/06-interfaces.md`, and `docs/09-current-status.md`. Execute or document the exact local demo path from a clean workspace: initialize, register three apps and one database, start with `routely`, inspect CLI status/logs, inspect dashboard status/logs, and stop workloads. Record date, OS, Node/npm, Docker availability, exact commands, expected/actual behavior, failures, and concrete Backend/Frontend bug instructions. Do not edit production code. Commit only the QA report if complete.

- [ ] Security: review local demo trust boundaries and write findings under `docs/security/`. Read `docs/07-security-and-risks.md` and `docs/03-demo-acceptance-plan.md`. Check local daemon/dashboard binding assumptions, same-origin dashboard API boundary, local command execution risk language, secret/log exposure, and untrusted text rendering in local logs/status. Write a concise report with findings, severity, reproduction or evidence, and remediation owners. Do not edit production code unless explicitly assigned. Commit only the security report if complete.

Human review after phase: Routely Lead/PM should confirm the local demo is ready before moving to VPS work.
