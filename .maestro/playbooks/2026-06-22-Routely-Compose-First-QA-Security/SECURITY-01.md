# Routely Compose-First Security Validation

Task-based playbook: each checkbox task must be executable in a fresh Security agent context. Security owns reports under `docs/security/` and must not commit reports. Report findings to Routely Lead only. Preserve unrelated dirty worktree changes and do not edit production code unless explicitly assigned later.

- [ ] Security: review the completed Compose-first Routely implementation and write a trust-boundary report. Read `AGENTS.md`, `docs/00-product-brief.md`, `docs/01-alpha-plan.md`, `docs/02-team-execution-plan.md`, `docs/03-demo-acceptance-plan.md`, `docs/05-architecture.md`, `docs/06-interfaces.md`, `docs/07-security-and-risks.md`, `.maestro/playbooks/2026-06-22-Routely-Compose-First-Execution/BACKEND-01.md`, `.maestro/playbooks/2026-06-22-Routely-Compose-First-Execution/FRONTEND-01.md`, and recent Compose-first commits before reviewing. Assess local daemon/dashboard boundaries, same-origin `/api/*` browser access, production admin-token auth, Docker/Compose command exposure, domain/proxy/TLS truthfulness, env/secrets redaction, backup file metadata and restore limitations, logs/deploy history exposure, health/metrics retention, GitHub webhook/delivery/branch trust assumptions, untrusted text rendering, outbound notification/webhook risks, and solo-operator/no-RBAC assumptions. Run targeted static checks and existing relevant tests where practical: `npm run lint`, `npm run test --workspace apps/cli`, `npm run test --workspace apps/web`, `node --check apps/daemon/src/server.js`, and focused greps/tests for raw secret leakage or unsafe rendering if useful. Create a dated report under `docs/security/` with YAML front matter, scope, evidence, severity, reproduction notes, affected owners, and remediation instructions. Do not edit production code. Do not commit the Security report.
